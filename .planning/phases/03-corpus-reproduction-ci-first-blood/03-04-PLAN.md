<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner -> external actions | GitHub Actions pulls third-party actions; unpinned tags are a supply-chain tampering vector. |
| committed corpus/manifest -> every-commit CI | CI re-validates the stored corpus from JSON on every push, making silent tampering detectable at merge time. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-04 | Tampering (supply chain) | astral-sh/setup-uv + actions/checkout | mitigate | Pin every action to a 40-char commit SHA (not a moving tag); `uv sync --locked` + `uv run --frozen` make the committed lockfile authoritative — no unverified auto-download. |
| T-3-03 | Elevation/Repudiation | trust root under optimized bytecode | mitigate | A dedicated `python -O -m pytest` job runs the raise-based verifier + R1 over the corpus, proving the trust check is not a stripped-assert no-op (extends test_verifier_dash_O.py). |
| T-3-02 | Tampering (accidental drift) | interpreter/env drift changing regenerated bytes | mitigate | Pinned 3.12.13 for required jobs + a non-blocking 3.13 canary (`continue-on-error`) that fails on purpose on set-order/random drift (R2 + fingerprint). |
| T-3-SC | Tampering (supply chain) | package installs | accept | No new PyPI packages in Phase 3; the runtime stack is the Phase-1 verified lockfile. slopcheck N/A (no installs); the only new dep is the SHA-pinned CI action. |
