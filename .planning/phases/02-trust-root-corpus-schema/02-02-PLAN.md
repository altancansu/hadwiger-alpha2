<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| record → store (append gate) | Every record crosses the verifier before any byte is written; unverified records are refused. |
| in-memory array → on-disk corpus | Written atomically (tempfile + os.replace); a reader/crash sees old-or-new, never a torn file. |
| new append → existing records | Existing records are immutable; the append-only prefix guard refuses any change to prior records. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | in-place edit of an existing corpus record | mitigate | Append-only prefix-immutability guard (json sort_keys byte-compare of the prefix) raises on any prior-record change; status stays a derived view (VRF-03, Phase 6). |
| T-02-05 | Denial / Tampering | crash mid-write corrupts the corpus anchor | mitigate | tempfile in path.parent → os.fsync → os.replace atomic same-filesystem swap; temp unlinked on failure. |
| T-02-07 | Elevation | a record enters the corpus unverified | mitigate | append_certificate mandatorily calls verify_model_record + verify_chi_witness and requires verified=True before appending. |
| T-02-03 | Repudiation | truncated / self-certifying family in schema | mitigate | schema stores the FULL family (len == had_2 >= chi) and refuses a short family; verifier re-checks k>=chi + all C(k,2) cross-adj at append. |
| T-02-SC | Tampering | package installs | accept | No package-manager installs this phase — schema/store are stdlib-only; existing pins from Phase 1's uv.lock (RESEARCH § Package Legitimacy Audit: nothing to slopcheck). |
</threat_model>
